# ❄️ Кондиционер — Температура v3.14

Логический сценарий для **SprutHub**. Управляет кондиционером по температуре в комнате — включает, выключает и плавно регулирует скорость вентилятора. Учитывает присутствие людей, открытые окна, время суток, уличную температуру и ручное управление.

> Сценарий работает с минимальным набором — **кондиционер + датчик температуры + пороги**. Все остальные датчики и переключатели опциональны: если не задан — условие считается выполненным автоматически.

## Принцип работы

Каждую минуту (и при изменении состояния кондиционера) сценарий проходит по цепочке проверок. Если условие не выполнено — кондиционер выключается и дальнейшие проверки не идут.

| # | Проверка | Что происходит |
|---|----------|----------------|
| 1 | **Зима / Лето** | Если переключатель «Зима» включён — вся логика остановлена |
| 2 | **Режим охраны** | «Не дома» → выключить. «Ночь» → ночные пороги. «Дома» → стандартный режим |
| 3 | **Открытые окна** | Хотя бы одно открыто → немедленное выключение |
| 4 | **Уличная температура** | Холоднее порога → не включать |
| 5 | **Присутствие дома** | Никого из жильцов дома → не включать |
| 6 | **Присутствие в комнате** | Никого в комнате → таймер задержки → выключить |
| 7 | **Температура и скорость** | Включить/выключить по порогам, пересчитать скорость |
| 8 | **Дисплей** | Днём ON, ночью OFF |

## Требования

- SprutHub с поддержкой логических сценариев (движок Nashorn/JVM)
- Кондиционер с интеграцией через **Thermostat-сервис** (`HC.TargetHeatingCoolingState`, `HC.CurrentHeatingCoolingState`, `HC.C_FanSpeed`)
- Внешний **датчик температуры** в комнате (`TemperatureSensor`)
- Работа протестирована на кондиционере Ballu, контроллер Lytko 102 (прошивка 250403.2 и WiFi 3.1.5+29)

## Установка

1. Скачать файл `Кондиционер_Температура_vх.хх.json
2. - Шаблон_устройства - MQTT_(Lytko)_102_v7.json - для прошивки **3.1.5+29** 
   - Шаблон устройства - MQTT (Lytko) + 102 + дисплей.json - для прошивки **250403.2**`
3. В SprutHub → Автоматизация → Логические сценарии → Импорт
4. Привязать сценарий к **Thermostat-сервису** кондиционера
5. Заполнить обязательные настройки (см. ниже)
6. Включить сценарий

> ⚠️  Для изминения параметров/датчиков необходимо **отключить** сценарий, внести изминения и снова активировать сценарий.

## Логика скорости и динамика

### 💨 Ступени скорости

Скорость пересчитывается **каждую минуту** по текущей температуре. При снижении температуры скорость уменьшается автоматически — без перезапуска кондиционера. Задаёте верхнюю границу для каждой ступени:

| Ступень | Название | Порог по умолчанию (день) |
|---------|----------|:-------------------------:|
| 1 | Тихо | < 26°C |
| 2 | Медленно | < 27°C |
| 3 | Средне | < 28°C |
| 4 | Быстро | < 29°C |
| 5 | Турбо | ≥ 29°C |

Отдельный набор порогов для **ночного режима** (те же 5 ступеней).

### 📈 Динамическое ускорение

| Параметр | По умолчанию | Описание |
|----------|:------------:|----------|
| **Динамика включена** | ON | Ползунок вкл/выкл |
| **Минут без снижения** | 15 | Если температура не падает N минут — скорость +1 |

- Если температура стагнирует (не снижается) N минут — скорость повышается на одну ступень (`boost +1`)
- `boost` держится пока температура остаётся в той же ступени скорости или выше
- Снижение скорости происходит только если `calcSpeed` вернул значение **ниже зоны накопления boost** — то есть температура реально ушла в другую ступень вниз
- Защита от скачков: новая (меньшая) скорость применяется только после подтверждения N циклов подряд

**Пример работы** при `speedTemp1=26`, `dynamicMinutes=15`, `speedDownDelay=2`:

```
T=26.1°C → calcSpeed=2 (медленно), boost=0, итог=2
... 15 минут без снижения ...
T=26.0°C → boost +1, итог=3 (средне), baseSpeedAtBoost=2
T=25.8°C → calcSpeed=1 (тихо) < baseSpeedAtBoost(2) → pending 1/2
T=25.7°C → calcSpeed=1 → pending 2/2 → снижение: 3→1 (сброс буста)
```

> Если температура колеблется около границы ступени (например 25.9/26.1°C), параметр **«Циклов подтверждения»** предотвращает постоянные переключения — скорость меняется только если новое значение устойчиво держится N минут подряд.

### 🌙 Ночной режим

| Параметр | По умолчанию | Описание |
|----------|:------------:|----------|
| **Начало ночного режима** | 23:00 | Формат ЧЧ:ММ |
| **Конец ночного режима** | 08:00 | Формат ЧЧ:ММ |

Ночной режим также активируется при режиме охраны «Ночь» (`SecuritySystemCurrentState = 2`).

### 🔒 Сигнализация (хаб СХ)

SecuritySystem-сервис. Если не задан — считается режим «Дома».

| Состояние | Значение | Поведение |
|-----------|:--------:|-----------|
| Дома | 0 | Стандартный режим |
| Не дома | 1 | Кондиционер выключен |
| Ночь | 2 | Ночные пороги и скорости |

### ❄️☀️ Переключатель Зима/Лето

Switch-сервис. **ON = зима** (логика выключена), **OFF = лето** (логика работает). Удобно отключить кондиционирование на зиму без удаления сценария. Если не задан — считается лето.

### 👥 Присутствие

| Параметр | Тип | Описание |
|----------|-----|----------|
| **Присутствие человека 1** | OccupancySensor | Персональный датчик жильца (напр. Андрей) |
| **Присутствие человека 2** | OccupancySensor | Персональный датчик жильца (напр. Олеся) |
| **Присутствие человека 3** | OccupancySensor | Третий жилец (опционально) |
| **ДП в комнате** | OccupancySensor | Датчик присутствия в данной комнате |

Кондиционер включается только если **хотя бы один** из персональных датчиков показывает «дома» **и** ДП в комнате зафиксировал присутствие. Если датчики не заданы — условие считается выполненным.

### ⏱️ Задержка выключения при уходе из комнаты

| Параметр | Тип | По умолчанию | Описание |
|----------|-----|:------------:|----------|
| **Задержка включена** | Boolean | ON | Ползунок вкл/выкл |
| **Задержка (мин)** | Integer | 5 | Сколько минут ждать после ухода |

- **Задержка ON** — при уходе запускается таймер, кондиционер выключается через N минут
- **Задержка OFF** — кондиционер выключается сразу при исчезновении присутствия
- Таймер запускается только если кондиционер **реально работает** в момент ухода

### 🪟 Датчики окон

До **4 датчиков** ContactSensor на комнату. Если хотя бы одно окно открыто — кондиционер немедленно выключается. При закрытии всех окон — возобновляет работу при выполнении остальных условий. Если датчики не заданы — считается что окна закрыты.

### 🌤️ Уличная температура

| Параметр | По умолчанию | Описание |
|----------|:------------:|----------|
| **Датчик уличной температуры** | — | TemperatureSensor (опционально) |
| **Минимальная уличная температура** | 15°C | Ниже этого — кондиционер не включается |

Если датчик не задан — проверка не выполняется.

### 🖥️ Дисплей кондиционера

Switch-сервис дисплея. **Днём** — ON, **ночью** — OFF. Если не задан — не управляется.

## Ручное управление

Сценарий распознаёт ручное выключение кондиционера и не включает его обратно автоматически в течение заданной паузы.

| Параметр | По умолчанию | Описание |
|----------|:------------:|----------|
| **Пауза после ручного выключения (мин)** | 30 | Сколько минут не трогать кондиционер после ручного выключения |

**Поведение:**
- Выключили вручную → сценарий ждёт N минут → возобновляет автоуправление
- `0` = ждать пока температура не опустится до порога выключения (tOff)
- Включили вручную в период паузы → пауза снимается, сценарий берёт управление

**Защита от MQTT-реверта** — кондиционеры с контролером Lytko через MQTT могут сообщать хабу своё физическое состояние и тем самым перезаписывать команду выключения. Сценарий обнаруживает это и повторяет команду выключения автоматически.

**Защита от «тихого» включения** — если сценарий отправил команду включения, но `CurrentHeatingCoolingState` остаётся `0` (устройство не откликнулось) — сценарий сбрасывает состояние и повторяет попытку в следующем цикле.

## Логирование и диагностика

| Параметр | Тип | По умолчанию | Описание |
|----------|-----|:------------:|----------|
| **Логирование событий** | Boolean | ON | Единый ползунок для всех сообщений в лог |

- **ON** — в лог пишутся все события: включение/выключение, изменение скорости, присутствие, окна, режимы, диагностика
- **OFF** — только критические ошибки (`console.error`) и предупреждения об offline-датчиках (`console.warn`)

При включённом логировании примеры сообщений:

```
[Гостиная] ON → целевая=24°C скорость=медленно T=26.1°C | было: выкл
[Гостиная] Снижение скорости: Т=25.3°C → 2→1 (сброс буста)
[Гостиная] Динамика +1: Т=26.5°C не снижалась 15 мин → скорость →3
[Гостиная] Выключение (Т=24.0°C). Причина: Т=24°C <= выкл. порог 24°C
[Гостиная] Внешнее выключение — блокируем автовключение
[Гостиная] MQTT-реверт после ручного выключения — блокируем
[Гостиная] Устройство не ответило на включение — сброс и повтор в этом цикле
[DEBUG] [Гостиная] Т=26.1°C tOn=26 tOff=25 ACstate=2 wantedOn=true lastMode=2 (день)
[DEBUG] [Гостиная] Т=26.1°C пороги=[26,27,28,29] база=медленно boost=0 итог=медленно (день)
[DEBUG] trigger: source=C[383.13.16 TargetHeatingCoolingState '2' Охлаждение] value=2
```
Ошибки крона пишутся с указанием комнаты и номера строки:
```
[Спальня] cron: ... (line 742)
```

## История версий

### v3.13
- Исправлена логика динамического ускорения: boost больше не сбрасывается через N минут после накопления. Снижение скорости происходит только если температура реально ушла в более низкую ступень (`calcSpeed < baseSpeedAtBoost`), а не просто из-за наличия boost

- Все `setValue()` обёрнуты в `Number()` — защита от ошибки `Undefined cannot be cast to class vy`
- Название комнаты и номер строки в сообщениях об ошибках крона

### v3.11
- Добавлен параметр **«Циклов подтверждения перед снижением скорости»** (Integer, по умолчанию 2) — защита от скачков скорости при колебании температуры около границы ступени
