# 🔧 Фильтр рекуператора VAKIO BASE SMART — логический сценарий для SprutHub

Логический сценарий для умного дома **SprutHub**, который отслеживает ресурс фильтра рекуператора **VAKIO BASE SMART** на основе реального объёма прокачанного воздуха, режима работы, скорости вентилятора, условий установки и сезонности.

Данные **персистентны** — сохраняются в характеристиках устройства и переживают перезапуск сценария и хаба.

---

## Зачем это нужно

Производитель рекомендует менять фильтр раз в 6 месяцев, но реальный ресурс сильно зависит от:
- режима работы (приток, вытяжка, рекуперация)
- скорости вентилятора
- этажа и расстояния до дороги
- сезона (весенняя пыльца, летняя пыль, зимний чистый воздух)

Сценарий считает **раздельный износ** двух слоёв фильтра:
- **Механика F7** — задерживает пыль, пыльцу, PM2.5
- **Угольный слой** — адсорбирует газы, выхлопы, запахи

Это важно: угольный слой исчерпывается быстрее вблизи оживлённой дороги, а механика F7 дольше служит на высоких этажах где крупных частиц меньше.

## Принцип расчёта

### Накопление объёма

Сценарий срабатывает **при каждом изменении режима или скорости** рекуператора. При каждом срабатывании:

1. Закрывается предыдущий интервал — считается объём `м³ = поток(м³/ч) × время(ч)`
2. Открывается новый интервал с текущим режимом и скоростью
3. Объём распределяется по режимам:
   - **Приток / Вытяжка** — весь объём в соответствующий счётчик
   - **Рекуперация** — 50% в приток, 50% в вытяжку (реверс каждые 40 сек по паспорту)

Страховочный **cron раз в час** фиксирует накопленное если режим не менялся долго (ночной режим, длительная рекуперация).

### Реальный ресурс фильтра

```
Реальный ресурс = Базовый ресурс ÷ (К_район × К_этаж × К_сезон)
```

Чем хуже условия — тем меньше реальный ресурс.

**Пример** для оживлённой дороги (50–150 м), 6 этаж, лето:
```
F7 реальный = 60 000 ÷ (1.25 × 1.00 × 1.05) ≈ 45 714 м³
Уголь реальный = 30 000 ÷ (1.25 × 1.00 × 1.05) ≈ 22 857 м³
```

### Износ по режимам работы

| Режим | Коэффициент F7 | Пояснение |
|---|:---:|---|
| Приток | 1.00 | Полная нагрузка — уличный воздух через фильтр |
| Приток MAX | 1.30 | Турбо-режим |
| Рекуперация лето | 1.00 | Объём уже ×0.5 при распределении |
| Рекуперация зима | 1.00 | Аналогично |
| Вытяжка | **0.00** | Наружный воздух через фильтр не проходит |
| Вытяжка MAX | **0.00** | Аналогично |
| Ночной | 0.50 | 5 мин вытяжка MAX + тихий приток |

**Угольный слой** изнашивается только на притоке — адсорбирует газы из уличного воздуха. На вытяжке уголь не расходуется.

### Персистентное хранение данных

Накопленные данные записываются в характеристики устройства после каждого интервала:

| Сервис устройства | Тип | Хранит |
|---|---|---|
| Объём воздуха | `C_GasMeter` | Всего м³ через фильтр |
| Приток м | `C_GasMeter` | Накопленный приток |
| Вытяжка м | `C_GasMeter` | Накопленная вытяжка |
| Рекуперация м | `C_GasMeter` | Накопленная рекуперация |
| Механика F7 | `FilterMaintenance` | Остаток ресурса F7 % |
| Угольный слой | `FilterMaintenance` | Остаток ресурса угля % |
| Фильтр общий | `FilterMaintenance` | Общий остаток + кнопка сброса |

При перезапуске сценария или хаба данные восстанавливаются из устройства автоматически.

## Установка

### Требования

- **SprutHub** с поддержкой логических сценариев
- Рекуператор **VAKIO BASE SMART** интегрированный через MQTT

### Шаг 1 — Импортировать шаблон устройства

Импортировать `VAKIO_Base_Smart_with_Filter_v3fix.json` как шаблон устройства в SprutHub.

> ⚠️ Важно: имена сервисов в шаблоне не должны содержать скобок.

### Шаг 2 — Импортировать логический сценарий

Импортировать `Filter_Monitor_v3.10.json` как логический сценарий.

### Шаг 3 — Настроить сценарий

В настройках сценария заполнить:

1. **Рекуператор (Fan-сервис)** — выбрать из списка Fan-сервис рекуператора VAKIO
2. **Дата установки фильтра** — формат `ДД-ММ-ГГГГ`, например `01-06-2025`
3. **Базовый ресурс F7** — выбрать из таблицы выше
4. **Базовый ресурс угля** — аналогично
5. **Тип района** — выбрать из списка
6. **Этаж** — номер этажа
7. **Производительность по скоростям** — заполнены паспортными значениями, менять не нужно

### Шаг 4 — Проверить работу

Включить **«Подробное логирование»** на первые несколько дней. В логе автоматизаций будут строки:

```
[Фильтр] [Гостиная / Рекуператор] Новый интервал: Приток скорость=3
[Фильтр] [Гостиная / Рекуператор] Интервал: Приток | скор.3 | 45 мин | +60.0 м³ | F7 +0.131% | Уголь +0.262%
[Фильтр STATUS] [Гостиная / Рекуператор] F7=0.13% | Уголь=0.26% | Приток=60.0м³ | ...
[Фильтр КОЭФФ] [Гостиная / Рекуператор] Район=1.25 | Этаж=1.00 | Сезон=1.05 | Делитель=1.313
[Фильтр] [Гостиная / Рекуператор] ♻️ Восстановлено из устройства: F7=0.13% Уголь=0.26% Всего=60.0м³
```

## Мониторинг

### Где смотреть данные

- **Карточка устройства** — сервисы «Механика F7», «Угольный слой», «Фильтр общий», счётчики м³
- **Лог автоматизаций** — строки `[Фильтр STATUS]` и `[Фильтр КОЭФФ]` при включённом логировании
- **Уведомления SprutHub** — стандартные уведомления при `FilterChangeIndication = 1`

### Когда менять фильтр

Сценарий устанавливает `FilterChangeIndication = 1` когда любой слой изношен на 85% и более. SprutHub генерирует стандартное уведомление.

> Угольный слой, как правило, исчерпывается раньше механики F7 вблизи оживлённых дорог.

### После замены фильтра

1. Физически заменить фильтр
2. Нажать кнопку **«Сброс фильтра»** в карточке устройства (сервис «Фильтр общий»)
3. Сценарий автоматически обнулит все счётчики

## Известные особенности SprutHub

- `variables` в логических сценариях **не персистентны** — сбрасываются при перезапуске сценария или хаба. Поэтому все данные хранятся в характеристиках устройства (`C_CubicMeter`).
- `HC.FilterLifeLevel` — характеристика **read-only** в стандарте HomeKit. `setValue` обновляет значение в текущей сессии для отображения, но не сохраняется в БД. Поэтому износ F7 и угля персистируется через `C_CubicMeter`.
- Имена сервисов в шаблоне устройства **не должны содержать скобок** — СХ некорректно обрабатывает их при `getName()`.
