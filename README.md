# 🏠 Логические сценарии для SprutHub

Коллекция продакшен-готовых логических сценариев для платформы умного дома **[Sprut Hub](https://spruthub.ru)**. Все сценарии написаны на JavaScript, протестированы в реальных условиях и развиваются итеративно — с детальным логированием, понятными настройками и историей версий.

## ⚙️ Общие требования

- **Sprut Hub** с поддержкой логических сценариев
- Устройства, интегрированные через стандартные HomeKit-сервисы (Thermostat, Fan, FilterMaintenance, CarbonDioxideSensor, AirQualitySensor, ContactSensor и т.д.)
- Для рекуператора — устройство **VAKIO BASE SMART**, подключённое по **MQTT**

## 🚀 Установка

Общий порядок для любого сценария:

1. Скачать `.json` сценария (и при необходимости — шаблон устройства) из соответствующей папки
2. В SprutHub → **Логика → Шаблоны** → импортировать
3. Привязать сценарий к нужному сервису устройства
4. Заполнить настройки (см. README конкретной папки)
5. Включить сценарий

## 🛠️ Особенности платформы SprutHub, учтённые в коде

При разработке учтён ряд платформенных ограничений Nashorn/SprutHub:

- JS-движок — без стрелочных функций, без `String.prototype.padStart` и т.п.
- `variables` логического сценария **не персистентны** — обнуляются при перезапуске; для хранения используются характеристики устройства (`C_GasMeter` / `C_CubicMeter`)
- `HC.FilterLifeLevel` — **read-only**, `setValue` не сохраняется между сессиями
- Поиск сервисов по имени ненадёжен — спецсимволы (`³`, скобки и др.) обрезаются при `getName()`, поэтому используется поиск по UUID
- `HS.FilterMaintenance` не может будить сценарий через `sourceServices`
- В UI SprutHub тип поля `"Number"` не поддерживается — нужно использовать `"Integer"` `"Double"`
- Функции, использующие `options`, должны получать его явным параметром (иначе падает на старте/триггере)

## 📝 Логирование

Каждый сценарий использует единый переключатель **«Логирование событий»**, объединяющий все информационные сообщения в один ползунок — без отдельных debug-флагов и лишних настроек в интерфейсе.
